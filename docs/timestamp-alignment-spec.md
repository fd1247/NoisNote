# 时间戳对齐功能需求规格说明

## 背景

NoisNote 当前主线 ASR 使用 Qwen3-ASR GGUF 完成本地转录，历史详情页已经提供“转录文本”和“总结内容”两个结果视图，模型管理页只管理正式 ASR 模型。近期热词能力已经接入转录配置和历史元数据，但转录结果仍只有整段文本，没有可用于定位音频位置的逐句时间轴，也不能导出字幕文件。

上游 HaujetZhao/Qwen3-ASR-GGUF 已提供 Qwen3-ForceAligner GGUF 辅助模型能力。根据本地 demo 代码与上游文档，启用方式是在 ASR 引擎配置中打开 `enable_aligner`，并为 `align_config` 指定 ForceAligner 模型目录及三个关键文件：

- `qwen3_aligner_encoder_frontend.int4.onnx`
- `qwen3_aligner_encoder_backend.int4.onnx`
- `qwen3_aligner_llm.q4_k.gguf`

本机已预置待验证模型目录：

```text
~/Documents/NoisNote/models/Qwen3-ForceAligner-0.6B-gguf
```

本需求要求先在 `demo/model_test` 下验证真实时间戳输出能力，再分步骤接入正式应用，并确保每一步都可通过自动测试或明确的手动验收验证。

## 目标

- 支持下载、识别和管理 Qwen3-ForceAligner GGUF 0.6B 时间戳对齐模型。
- 在通用设置中提供“启用时间戳”选项，转录时根据用户设置和模型完整性决定是否启用。
- 在具备时间戳结果的历史记录中展示“逐句时间轴”标签页。
- 支持从转录结果导出 txt、srt 和 markdown 三种格式。
- 保持时间戳模型作为辅助能力，不进入 ASR 模型下拉列表，不影响用户选择主 ASR 模型。

## 功能需求

- F1: 模型下载管理应支持时间戳对齐模型 `luciacx/Qwen3-ForceAligner-GGUF-0.6B-mixed`。
- F2: 时间戳对齐模型应出现在模型管理页的下载/已下载管理中，但不应出现在通用设置的 ASR 模型选择列表中。
- F3: 通用设置页应在 ASR 模型附近提供“启用时间戳”开关，并提示启用该功能需要下载时间戳对齐模型。
- F4: 转录开始时应同时检查用户开关和时间戳模型完整性；只有二者都满足时才启用时间戳对齐。
- F5: 如果用户启用了时间戳但模型未下载或不完整，本次转录应降级为普通转录，并给出可见提示或记录可诊断原因，不应让转录失败。
- F6: 启用时间戳的转录结果应保存结构化时间轴数据，至少包含每个片段的开始时间、结束时间和文本内容。
- F7: 历史详情页应在“转录文本”后添加“逐句时间轴”标签页；只有当前历史记录包含可用时间轴数据时显示该标签页。
- F8: “逐句时间轴”视图应按时间顺序展示片段，用户能直接阅读每句文本及其时间范围。
- F9: 转录结果应支持导出格式选择：txt 导出原始转录文字，srt 导出字幕，markdown 导出总结结果。
- F10: srt 导出仅在当前历史记录存在时间轴数据时可用；缺失时间轴时应禁用该格式。
- F11: 重新转录时应清理旧的时间轴与旧的导出结果，避免历史记录显示过期字幕。
- F12: 历史元数据应记录本次转录是否请求时间戳、是否实际启用、使用的对齐模型、降级原因和对齐性能摘要。

## 非功能需求

- N1: 自动化测试不得运行真实大模型推理；真实 ForceAligner 推理仅作为 `demo/model_test` 下的手动验证步骤。
- N2: 时间戳模型完整性校验应与现有 GGUF 模型校验保持一致，必须检查关键文件，不依赖配置里手写路径是否可信。
- N3: 时间戳功能失败不应影响普通转录成功；降级原因需要进入日志或历史元数据，便于排查。
- N4: UI 变更应保持当前浅色主界面和设置页风格，新增标签、导出入口和开关不应挤压现有操作。
- N5: 时间轴存储格式应稳定、可扩展，后续可支持点击跳转音频播放，但本阶段不要求实现播放。
- N6: 导出文件应写入当前历史记录目录或由系统文件保存对话框选择目标，不应写入用户数据目录之外的隐式位置。

## 前置验证要求

- V0.1: 在 `demo/model_test` 下新增或复用一个最小验证脚本，读取本地 ASR 模型和 `~/Documents/NoisNote/models/Qwen3-ForceAligner-0.6B-gguf`，启用 `enable_aligner` 完成一段短音频转录。
- V0.2: 验证脚本应输出三类文件：原始转录文本、结构化时间戳 JSON、SRT 字幕。
- V0.3: 验证结果应证明 `result.alignment.items` 非空，且每一项包含单调递增的 `start_time` 与 `end_time`。
- V0.4: 验证脚本应记录 CPU 与当前设备策略下的耗时，形成后续接入时的性能基线。

## 分阶段实现要求

### 阶段一：模型清单与配置

- 将 ForceAligner 加入模型管理清单，标记为辅助模型。
- 增加 `enable_timestamps` 配置默认值，默认关闭。
- 通用设置页显示时间戳开关和下载提示。
- 验证：模型管理测试能看到辅助模型；ASR 模型下拉测试确认不包含辅助模型；配置合并测试确认默认值存在。

### 阶段二：Runtime 接入

- 扩展转录运行配置，携带是否请求时间戳、对齐模型目录和对齐模型文件名。
- 当开关打开且模型完整时，构造带 `enable_aligner=True` 的 ASR 引擎配置。
- 解析底层 `result.alignment.items`，转成应用内部时间轴数据。
- 验证：使用 fake runtime 或 monkeypatch 的单元测试覆盖启用、未启用、模型缺失降级三种路径。

### 阶段三：历史存储与 UI

- 在历史记录目录中保存时间轴文件，例如 `timeline.json`。
- 扩展历史记录扫描结果，能判断当前记录是否有时间轴。
- 历史详情页新增“逐句时间轴”标签，仅在有数据时显示。
- 验证：历史服务测试覆盖保存、读取、清理；Qt UI 测试覆盖有/无时间轴时标签显示差异。

### 阶段四：导出

- 增加统一导出入口，允许选择 txt、srt、markdown。
- txt 使用当前转录文本；srt 使用时间轴数据生成字幕；markdown 使用当前总结内容。
- 导出成功后更新状态提示和必要的历史元数据。
- 验证：服务层测试覆盖三种导出内容；UI 层测试覆盖缺失时间轴时 srt 不可导出。

## 不做的事情

- 不把 ForceAligner 当作独立 ASR 模型供用户选择。
- 不在本阶段实现音频播放器、点击时间轴跳转、波形视图或逐字高亮播放。
- 不为缺失时间戳的旧历史记录补跑对齐任务。
- 不在自动化测试中下载模型或运行真实 GGUF 推理。
- 不引入除现有 GGUF runtime 所需之外的新模型后端。

## 验收标准

- AC1: 模型管理页可下载并识别 `Qwen3-ForceAligner-GGUF-0.6B`，已下载状态以三个关键文件完整为准。
- AC2: 通用设置 ASR 模型下拉只显示正式 ASR 模型，不出现 ForceAligner。
- AC3: 通用设置中可以开启或关闭时间戳，保存后配置文件包含对应布尔值。
- AC4: 用户开启时间戳且 ForceAligner 完整时，转录元数据记录 `timestamps.enabled=true`，历史记录生成时间轴数据。
- AC5: 用户开启时间戳但 ForceAligner 缺失时，转录仍成功，元数据记录 `timestamps.enabled=false` 和降级原因。
- AC6: 没有时间轴数据的历史记录只显示“转录文本”和“总结内容”；有时间轴数据时显示“转录文本”“逐句时间轴”“总结内容”。
- AC7: 逐句时间轴页面按开始时间升序展示时间范围和文本，空文本片段不会显示为有效句子。
- AC8: txt 导出文件内容等于原始转录文字。
- AC9: srt 导出文件符合 SRT 编号和时间范围格式，且内容来自时间轴片段。
- AC10: markdown 导出文件内容来自总结结果；无总结时给出明确提示或导出空状态说明。
- AC11: 重新转录会清理旧时间轴与旧导出结果，新结果不会混入旧字幕。
- AC12: 自动化测试覆盖配置、模型清单、runtime 决策、历史存储、标签显示和导出格式；真实时间戳推理通过 `demo/model_test` 手动验证记录证据。
